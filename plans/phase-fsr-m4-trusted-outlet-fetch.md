# Phase FSR-M4: Trusted-outlet top-stories fetch

**Milestone:** M4 — Trusted-outlet top-stories fetch (PRD `plans/prd.md` §Milestones). Replaces keyword-first news fetch with a fetch keyed on **category + a curated ~10–15-domain authority set per category** (GDELT DOC `domainis:` query construction or GKG `SourceCommonName ∈ set`), preserving per-cell resilience + bounded gap-fill.
**Status:** Not started
**Estimated effort:** L

## Goal
When this phase is done, news ingestion can fetch the day's biggest stories per category from a curated authority-domain set — a pure `domainis:` query builder (net-new on `gdelt_doc.py`, which has NO domain filter today) and an additive GKG `SourceCommonName ∈ set` predicate both emit a correct domain-scoped query, the curated domain set is a lint-validated data file, and the pipeline can run keyed on category+domains (replacing keyword-first fetch for news) with one cell's failure skipping only that cell and an under-filled cell triggering a bounded window-widening gap-fill — **all unit-tested offline against mocked adapter responses and fixtures**. The real GDELT pull is the explicit LIVE-E2E residual.

## Why N=1 (single phase)
M4 is one shippable slice: a fetch path keyed on category+domains. Its four natural units (the domain data asset, the DOC query builder, the GKG predicate, the pipeline rekey) are one file-region each, all offline-verifiable, and the pipeline rekey (SP4) depends on the other three — so they belong in one `/run-phase` session with an internal dependency edge, not separate phases. N>1 would split the consumer (SP4) from its dependencies (SP1–SP3) across sessions for no benefit.

## Sub-phases

### Sub-phase 1: Curate the authority-domain data file + lint test  — INDEPENDENT
- **Files touched:** `agents/ingestion/authority_domains.py` (new — data + a pure `domains_for_category(category) -> list[str]` accessor), `tests/agents/ingestion/test_authority_domains.py` (new).
- **What ships:** A version-controlled, category-keyed authority-domain set: for each of the 8 topic categories in `agents/pipeline/categories.py::TOPIC_CATEGORIES` (`ai, geopolitics, business, environment, politics, tech, sport, arts`), an ordered list of ~10–15 bare outlet domains (e.g. `reuters.com`, `apnews.com`, `bbc.co.uk`) the fetch trusts. Exposed via a pure accessor `domains_for_category(category)` plus the raw mapping. Domains are stored lowercase, bare-host (no scheme/path), the form GDELT's `domain`/`SourceCommonName` uses (`_article_to_candidate` lowercases `domain`; GKG `outlet = LOWER(SourceCommonName)`).
- **Definition of done:** `test_authority_domains.py` passes and asserts the data contract — *every* `TOPIC_CATEGORIES` key is present (no category silently missing → fail loud); each list has between 8 and 20 domains (the ~10–15 target with slack); every domain matches a strict host regex `^[a-z0-9.-]+\.[a-z]{2,}$` (lowercase, bare, no scheme/path/uppercase — a malformed entry like `https://Reuters.com/world` fails the test); no duplicate domain within a category; `domains_for_category` returns the list for a known category and raises (not silently `[]`) for an unknown category. The test must fail if a category is dropped, a domain is mistyped, or the accessor swallows an unknown category — it encodes WHY (a missing/mistyped domain means that category fetches from the wrong/zero outlets, the catalog-quality risk).
- **Dependencies:** none.

### Sub-phase 2: Build the DOC `domainis:` trusted-domain query builder  — INDEPENDENT
- **Files touched:** `agents/ingestion/adapters/gdelt_doc.py` (add a pure `build_domain_query(domains, *, search_query=None) -> str` module function + wire an optional `domains` kwarg through `search`), `tests/agents/ingestion/test_gdelt_adapter.py` (extend).
- **What ships:** A net-new pure string builder that emits a GDELT DOC 2.0 trusted-domain query. For a domain set it produces a `(domainis:reuters.com OR domainis:apnews.com OR …)` clause (DOC has no native multi-domain param — it is OR-composed `domainis:` atoms); a single domain emits one bare `domainis:reuters.com` (no needless parens); given an optional category/keyword search string it AND-composes it with the domain clause. `search()` accepts a `domains: list[str] | None` kwarg and routes through the builder when present, leaving the existing keyword path byte-for-byte unchanged when absent (surgical — Rule 3). No network: the builder is a pure function; the throttle/backoff path in `_throttled_get` is unchanged and reused.
- **Definition of done:** Extended `test_gdelt_adapter.py` passes and asserts, on the **pure builder** (no HTTP): a multi-domain set yields a clause containing `domainis:` for *each* domain joined by ` OR ` and wrapped in parens; a single domain yields exactly `domainis:<d>` with no parens; order is preserved/deterministic; an empty domain list raises (fail loud — never emit an unfiltered query that would pull randoms); domains are emitted verbatim (already-normalized by SP1). One `search()` test with the existing `_FakeResponse`/mock-transport harness asserts that passing `domains=[...]` causes the outgoing `params["query"]` to contain the `domainis:` clause (response PARSING reuses the existing recorded-fixture article path — no live call). The builder test fails if `domainis:` is dropped, the OR-join breaks, or an empty set emits a query — it encodes WHY (no `domainis:` ⇒ fetch is no longer domain-scoped ⇒ randoms return, defeating M4).
- **Dependencies:** none for the builder; the `search()` integration test may import SP1's `domains_for_category` for realistic inputs but does not require it (a literal list works).

### Sub-phase 3: Add the GKG `SourceCommonName ∈ set` domain-filter predicate  — INDEPENDENT
- **Files touched:** `agents/ingestion/adapters/gdelt_bigquery.py` (add a pure `build_domain_filter_sql(param_name)` helper + an optional `domains` path on `search`/`search_active_interests` that injects a `LOWER(SourceCommonName) IN UNNEST(@domains)` predicate into the `raw` CTE via a parameterized array), `tests/agents/ingestion/test_gdelt_bigquery_adapter.py` (extend).
- **What ships:** An additive, parameterized domain-restriction path for the GKG adapter so the alternate fetch (GKG `SourceCommonName ∈ curated set`) is available alongside the DOC path. The domain set arrives as a bound `ARRAY<STRING>` query parameter (injection-safe, like the existing `@interest_terms` STRUCT array) and the SQL gains a `LOWER(SourceCommonName) IN UNNEST(@domains)` filter when domains are supplied; with no domains the SQL is **identical** to today (surgical). This sub-phase does NOT add `V2Themes` to the SELECT — that is M2's surface; this is only the domain predicate, kept orthogonal to avoid an M2/M4 file-collision (flagged in Risk lens).
- **Definition of done:** Extended `test_gdelt_bigquery_adapter.py` passes and asserts, **without BigQuery** (string-level SQL assertions, the module's existing offline style): when a domain set is supplied, the emitted SQL contains `LOWER(SourceCommonName) IN UNNEST(@domains)` and the job config binds an `ArrayQueryParameter("domains", "STRING", [...])` with the lowercased domains; when no domains are supplied the SQL string equals the current `_BATCH_SQL` (no spurious predicate — proves the path is additive/reversible); the domain array is lowercased before binding. The test fails if the predicate is missing when domains are given or present when they are not — it encodes WHY (the predicate is what makes the GKG path trusted-outlet-only).
- **Dependencies:** none.

### Sub-phase 4: Rekey ingestion to category+domains with per-cell resilience + bounded gap-fill  — SEQUENTIAL (needs 1, 2, 3)
- **Files touched:** `agents/ingestion/interest_keyed_pipeline.py` (add a `ingest_trusted_outlets(...)` entry that fans out **one fetch per (category, domain-set) cell** using SP1's `domains_for_category` + SP2/SP3 builders, replacing keyword-first fetch for news; preserve the existing per-cell try/except resilience shape and add bounded gap-fill), `tests/agents/ingestion/test_interest_keyed_pipeline.py` (extend with a category-keyed fake adapter).
- **What ships:** The fetch is keyed on **category + trusted domains** rather than narrow interest keywords: for each topic category the pipeline asks the adapter for that category's curated domains (via the SP2 DOC builder or SP3 GKG predicate), clusters into the canonical pool (reusing `StoryClusterer`), and tags. Resilience is preserved per-cell: one category's fetch raising `AdapterFetchError` skips **only that category**, never the batch (mirrors the existing `failed_interests` loop). Bounded gap-fill: when a category cell returns fewer than a floor `min_stories_per_category`, the pipeline retries that cell once with a **widened time window** (e.g. `since` pushed back by a bounded delta, still ≤ the DOC 3d / GKG partition ceiling) and logs an `under_filled` warning if still short (fail loud, never silent). The function stays pure over injected inputs (category list, domain accessor, adapter, clock) so it is fully unit-testable with a fake adapter and no network.
- **Definition of done:** Extended `test_interest_keyed_pipeline.py` passes with a **category-keyed fake adapter** (no network) and asserts: (a) a multi-category run fetches each category's cell and the canonical pool contains stories from the injected fixture domains for each category; (b) **per-cell resilience** — a fake adapter that raises `AdapterFetchError` for ONE category yields a result with the *other* categories' stories present and a `failed`-cell count of 1 (the batch did not abort); (c) **bounded gap-fill** — a category whose first fetch returns `< min_stories_per_category` triggers exactly one widened-window re-fetch (assert the second call's `since` is earlier by the bounded delta) and, if still short, emits the `under_filled` log/return-field rather than crashing; (d) gap-fill is bounded — it never re-fetches more than once per cell (no unbounded loop). Each assertion encodes WHY (resilience: one bad outlet must not blank the feed; gap-fill: a thin category must widen, not silently under-deliver — the "is that really all?" failure mode). The `assign_category`/theme-tagging path is **not** touched here (M2's surface).
- **Dependencies:** Sub-phase 1 (domain accessor), Sub-phase 2 (DOC builder), Sub-phase 3 (GKG predicate — the alternate path the rekey may route through).

## Phase-level definition of done
`/run-phase` validates, fully offline (no GDELT/DB — Rule 12, no "works end-to-end" claim from this sandbox):
1. `pytest tests/agents/ingestion/test_authority_domains.py tests/agents/ingestion/test_gdelt_adapter.py tests/agents/ingestion/test_gdelt_bigquery_adapter.py tests/agents/ingestion/test_interest_keyed_pipeline.py` all pass.
2. The curated authority-domain data file exists, covers all 8 `TOPIC_CATEGORIES`, and passes its lint/schema test.
3. The DOC `domainis:` builder is a pure function whose tests prove the emitted query contains `domainis:` for every domain and refuses an empty set; the GKG path injects `LOWER(SourceCommonName) IN UNNEST(@domains)` only when domains are supplied (else SQL unchanged).
4. The pipeline rekey fetches per (category, domain-set) cell, keeps per-cell resilience (one cell fails ⇒ batch survives), and performs exactly one bounded window-widening gap-fill on an under-filled cell, logging `under_filled` if still short.
5. No keyword-first news-fetch behavior regressed (the existing `ingest_active_interests` tests still pass — the rekey is additive, not a delete).

**LIVE-E2E (deferred):** a real GDELT pull (DOC `domainis:` or GKG `SourceCommonName ∈ set`) returning the day's big stories from the curated domains per category. Sandbox cannot hit GDELT (403) or the DB — marked deferred, not done.

## Out of scope
- **Theme→category mapping and adding `V2Themes` to the GKG SELECT** — that is **M2**. SP3 adds ONLY the domain predicate, not the theme SELECT, to avoid an M2/M4 collision on `gdelt_bigquery.py`'s `_BATCH_SQL`.
- **Authority-weighted importance / source-tier table / `min(1, outlet_count/12)` retirement** — that is **M3** (E1). M4 fetches from trusted domains; M4 does not score them.
- **Picking the final ~10–15 domains as an editorial/content-ops decision** — SP1 ships a defensible launch seed; the *authoritative* per-category curation (and bias diversity) is a content-ops tuning task, not a planning gate (PRD "open items").
- **Deleting `interest_keyed_pipeline.ingest_active_interests`** — kept (a later hybrid stays possible per Decision #1); the rekey is an additive trusted-outlet entry, not a removal.
- **Any real network/DB call.** LIVE-E2E only.

## Open questions
- **DOC vs GKG as the production fetch path:** PRD Decision #4 *prefers* DOC `domainis:`. Both are built here (SP2 + SP3) so `/run-phase` is not blocked, and SP4 routes through DOC by default with GKG available as the alternate. If `/run-phase` wants a single path, confirm DOC-default. (Non-blocking — both are offline-testable.)
- **`min_stories_per_category` floor and the widen-delta** for gap-fill — proposed: floor ≈ 5, widen by +1 day (DOC `timespan` 1d→2d, capped at the 3d ceiling). A tuning value, pinnable in `/run-phase`; the *mechanism* (one bounded widen + fail-loud log) is the contract, not the constant.
- **Cross-milestone data sharing (see below):** does the authority-domain set in SP1 share a source-of-truth with M3's source-tier *authority weights*? They are related concepts; this plan keeps them as **separate files** (M4 = which domains to fetch; M3 = how much each outlet's authority lifts importance) to avoid a cross-milestone file-collision, and flags the future consolidation.

SP1 (domains), SP2 (DOC builder), SP3 (GKG predicate), SP4 (rekey + resilience + gap-fill) collectively deliver exactly that — every clause maps to a sub-phase. No clause is unmapped.

**Risk lens (what blows up):** PASS with findings addressed.

SP4 *imports* SP1–SP3 but does not *edit* their files — so no two sub-phases write the same file. The only true ordering edge is SP4-after-{1,2,3}, marked SEQUENTIAL.

- *Test coverage (Rule 9):* each sub-phase DoD carries a test that fails on a business-rule break (missing `domainis:`, dropped category, no gap-fill widen) — not just compile checks. The resilience and gap-fill tests encode the user-facing "don't blank the feed / don't under-deliver" intent.
- *Reversibility:* no DB migration, no public API, no data deletion in this phase — the domain data file and the additive query paths are all reversible. `ingest_active_interests` is preserved, so the keyword path is not destructively removed.
- *Painting-into-a-corner (1→2→3→4 sim):* SP1 ships the domain accessor; SP2/SP3 build query emitters that take a domain list (independent of SP1's internals); SP4 wires accessor→emitter→clusterer. Executing 1→2→3 leaves exactly the inputs SP4 needs; SP4 still works. No corner.

**Irreversible sub-phases:** none. (No migration, no public-API change, no data deletion; `ingest_active_interests` retained.)
