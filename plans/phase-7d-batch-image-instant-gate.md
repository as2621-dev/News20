# Phase 7d: Gemini Batch image path + 5am instant-fallback gate — DEFERRED

**Milestone:** M7 — Production feed automation & first-run onboarding feed
**Status:** Deferred (owner call 2026-06-16 — ship instant-only first; build this after the midnight trigger is live and proven)
**Estimated effort:** L

## Goal
Story posters are generated by a cheaper **Gemini Batch** submission at pipeline time (text/audio/feed don't wait on it), attached as the batch completes, and any poster still missing by **05:00 ET** is regenerated **synchronously** ("Nano Banana Pro instant") so every morning's feed is image-complete.

## Why deferred
The current poster path (`orchestrate_story` → synchronous `gemini-3-pro-image-preview`, non-fatal, writes `story_ambient_poster_url`) already works and degrades gracefully. Phases 7/7b/7c ship the automation on this synchronous path. Batch is a cost/latency optimization, not a correctness requirement — sequenced after the trigger is live.

## Sub-phases

### Sub-phase 1: Validate + add a Gemini Batch image adapter
- **Files touched:** new `agents/m0/batch_image_adapter.py` (+ test)
- **What ships:** a pre-flight that confirms the Gemini **Batch API** supports `gemini-3-pro-image-preview`, and an adapter that submits a list of poster prompts as one batch job and returns a batch handle. If batch does not support the image model, stop and escalate (do not fake it).
- **Definition of done:** pytest (Gemini client mocked) — submitting N prompts creates one batch job and returns its id; the pre-flight capability check has an explicit pass/fail path.
- **Dependencies:** none

### Sub-phase 2: Pipeline submits posters to batch + `poster_status` column
- **Files touched:** `agents/pipeline/orchestrator.py`, `agents/pipeline/persist.py`, new migration `supabase/migrations/00NN_poster_status.sql`
- **What ships:** the daily run submits poster prompts to the batch adapter while text/audio/feed persist immediately; a new `poster_status` column (`pending|ready|failed`) on the story/digest makes readiness queryable instead of inferred from a NULL URL.
- **Definition of done:** pytest — a produced story persists with `poster_status='pending'` and a NULL poster URL when batch is used; existing synchronous path still sets `ready`. Migration applies cleanly on a scratch DB.
- **Dependencies:** Sub-phase 1
- ⚠ irreversible (schema migration)

### Sub-phase 3: Poster-attach poller
- **Files touched:** new `agents/pipeline/stages/poster_attach.py` (+ test), worker/trigger glue
- **What ships:** reads completed batch results and writes `story_ambient_poster_url`/`digest_ambient_poster_url` + `poster_status='ready'` (or `failed`).
- **Definition of done:** pytest — a completed batch result attaches the poster URL and flips status to `ready`; a failed item flips to `failed` without crashing the others.
- **Dependencies:** Sub-phase 2

### Sub-phase 4: 05:00 ET instant-fallback gate
- **Files touched:** `trigger/feedReadinessCheck.ts` (extend Phase 7c's cron) or new `trigger/posterInstantGate.ts`, worker endpoint glue
- **What ships:** at 05:00 ET, any current story with `poster_status != 'ready'` is regenerated via the **synchronous** poster path and attached; the cron also confirms stories/feeds were processed and alerts on a miss.
- **Definition of done:** unit test — a story left `pending` at 05:00 triggers one synchronous regeneration and ends `ready`; an already-`ready` story is skipped; a hard failure raises a structured alert event.
- **Dependencies:** Sub-phase 3 (and Phase 7c Sub-phase 3, which it extends)

## Phase-level definition of done
A midnight run submits posters to batch and finishes text/audio/feed without waiting; posters attach as the batch completes; by 05:00 ET every current story is `poster_status='ready'` (late ones regenerated synchronously), and the readiness cron alerts if not.

## Out of scope
- Everything in Phases 7 / 7b / 7c (prerequisites).
- Changing the poster *prompt/quality* pipeline (concept extraction, SERP seeding, grading) — only the submission mode changes.
