# Phase 2b: Grounded Interrogation — in-context Q&A + verification + composer

**Milestone:** M2 — Detail View + trust + interrogation
**Status:** Not started
**Estimated effort:** M (was L — dropped the vector-store layer)

## Goal
A user types a question into the Detail search box and gets a **source-grounded** answer with citation chips — or a clean `⌀ CAN'T ANSWER FROM SOURCE` refusal when off-source — powered by loading the active story's (small) grounding corpus **directly into the LLM context** plus a verification stage. This grounded answer API is the shared brain M3's voice mode will later reuse.

## Re-scope context (2026-05-31): no vector DB — load the corpus into context
The original plan ported TLDW's RAG stack (`agents/rag/*` → Pinecone: chunk → embed → vector-retrieve top-K). That made sense for TLDW, which grounds Q&A over a **large** corpus (full podcast transcripts / a whole multi-source briefing). News20's grounding scope is the **opposite**: explicitly **per-story, single-source, "<100s read"**. The whole per-story corpus is `detail_chunks` (a handful of short paragraphs — `s1` has **3**), `story_timeline` (a few one-line events), the ~140-word digest, and optionally the single extracted source body (Decision #4) — together ~300–2,000 words / a few thousand tokens. There is nothing to "retrieve from": vector search to pick the relevant passage out of *three* is pure overhead.

**Decision:** drop Pinecone + the embedder + chunker + retriever + the vector index. Load the entire per-story corpus into the prompt, constrain the model to answer **only** from it, and gate with the verification stage. This is cleaner, removes an infra dependency + a prereq, makes the zero-hallucination guarantee *easier* (the model sees the whole source; no retrieval-miss failure mode), and pairs with prompt/context caching for cheap repeat Q&A. The endpoint contract is **unchanged** (`QuestionAnswer { answer_text, answer_citations[], answer_is_grounded }`), so M3 voice reuse and the SP3/SP4 frontend/cache are unaffected. **Escape hatch:** if a story's corpus ever grows large (long multi-source bodies, archive search), reintroduce retrieval *behind the same endpoint contract* — an internal swap, not a redesign.

> **⚠ Prerequisites (Rule 12):** (a) **Phase 2** ships the `StoryDetail` shell that the composer mounts into (Phase 2 is in flight; 2b runs after its commit). (b) A **server endpoint** to run the LLM call + verification server-side — the static-export SPA cannot hold the LLM key or run verification client-side. This is the Python worker (FastAPI → Vercel/Railway/Fly, `prototype-port-map.md` §6) **or** a serverless function; stand up a minimal one in SP2 if M1 didn't. **No Pinecone / vector store is required.** (c) The active story's **grounding corpus text** must be seeded — `detail_chunks` is already seeded by M1 (Phase 1b) and is sufficient for the M2 demo.

## Sub-phases

### Sub-phase 1: Load a story's grounding corpus into a citeable context block (no vector DB)
- **Files touched:** `agents/qa/corpus.py` (**NEW** — `load_grounding_corpus(story_id)`), `agents/qa/models.py` (**NEW** — `GroundingCorpus`, `GroundingPassage`, `CitationTarget` typed models), `agents/shared/{settings,logger,exceptions}.py` (**PORT** if not already present from M1 — they are).
- **What ships:** A function that reads a single story's grounding text via Supabase-direct reads — `detail_chunks` (ordered by `chunk_index`), `story_timeline` (when/what), the digest script, and any available single-source body — and assembles a **labeled, citeable context block**: each passage tagged with a stable passage id + its source outlet, plus a citation manifest from `story_sources` (`source_outlet_name` + `source_article_url`, the citation-chip targets). No chunking/embedding/vector upsert — the whole corpus is small enough for one prompt.
- **Definition of done:** `load_grounding_corpus('s1')` returns a `GroundingCorpus` containing **all** of `s1`'s `detail_chunks` in `chunk_index` order, each as a labeled `GroundingPassage`, plus its `story_sources` citation targets; the assembled block is **bounded** (assert a char/token budget) and contains **only** that story's text — the test fails if another story's chunk ever appears (Rule 9; the per-story scope is now trivially enforced by the `story_id`-filtered read). Pytest mocks the Supabase client at the boundary (CLAUDE.md mocking) and asserts (i) chunk ordering, (ii) the citation manifest shape, (iii) the returned models are validated Pydantic. `ruff check` + `ruff format` pass.
- **Dependencies:** none within phase. Prereq: grounding corpus seeded (`detail_chunks` from M1 Phase 1b).

### Sub-phase 2: Grounded answer endpoint + verification + refusal contract
- **Files touched:** `agents/qa/{agent,prompts,models}.py` (**ADAPT** from TLDW `agents/chat/*` — ground on the in-context corpus, not a retriever), `agents/pipeline/stages/verification.py` (**PORT**), `agents/worker/main.py` (add route `POST /api/story/{story_id}/question`).
- **What ships:** The grounded Q&A endpoint: `load_grounding_corpus` (SP1) → answer **constrained to the in-context corpus** (system prompt forbids answering without the provided context, `prototype-port-map.md` §7) → `verification` stage gates the claim against the corpus text → returns `QuestionAnswer { answer_text, answer_citations[], answer_is_grounded }` per `api-contracts.md`. Citations map to the passage ids / `story_sources` outlets the answer used. Off-source / unsupported → `answer_is_grounded = false` + the refusal payload. The per-story corpus block is **prompt/context-cached** (Claude or Gemini context caching) so repeat questions about the same story are cheap. Failures return HTTP 200 with a graceful fallback (never breaks the conversation) and log a typed `ErrorResponse` with `fix_suggestion`.
- **Definition of done:** An on-topic question (e.g. "Why does Hormuz matter?" for `s1`) returns `answer_is_grounded = true` with ≥1 citation whose outlet/`source_url` traces to that story's `story_sources` / corpus passages; an off-source question ("what's the weather?") returns `answer_is_grounded = false` with **no fabricated answer**. Pytest mocks the **LLM only** (no retriever to mock now) and asserts the grounded vs refusal branch **and** citation provenance — the test fails if an ungrounded answer is ever surfaced as grounded (Rule 9; the brief's zero-tolerance accuracy guardrail, Open Q5 / Decision #5). `ruff` passes.
- **Dependencies:** Sub-phase 1. **⚠ This is the riskiest surface in M2** — the hallucination guardrail. Consider spiking it first if de-risking early.

### Sub-phase 3: Q&A frontend — composer + thread + suggested chips + citation/refusal contract
- **Files touched:** `src/components/detail/QaComposer.tsx`, `src/components/detail/QaThread.tsx`, `src/components/detail/SuggestedQuestionChips.tsx`, `src/lib/qa/askQuestion.ts`, `src/components/detail/StoryDetail.tsx` (mount the composer — Phase 2's file, edited here only; Phase 2b runs after Phase 2 completes, so no parallel conflict).
- **What ships:** The pinned bottom `QaComposer`; the `QaThread` rendering the `.dot-typing` thinking state → a grounded `.qa-bubble-a` with one `.cite-chip` per `answer_citations` entry, **or** the `.qa-refusal` blush card with the mono header `⌀ CAN'T ANSWER FROM SOURCE`; and `SuggestedQuestionChips` from `suggested_questions`. `askQuestion()` calls the SP2 endpoint and maps `answer_is_grounded` to the two visual states **byte-for-byte** (`prototype-port-map.md` §7 — this visual distinction is how users learn to trust the system; do not redesign it).
- **Definition of done:** Typing or tapping a suggested question shows the thinking state, then a grounded bubble with citation chips for an on-topic question, or the refusal card for an off-source one. Manual UI smoke + a component test asserting `answer_is_grounded = false` renders the refusal card and **never** an answer bubble (Rule 9).
- **Dependencies:** Sub-phase 2; Phase 2 SP2 (the `StoryDetail` shell). (Unchanged by the re-scope — purely a frontend consumer of the SP2 contract.)

### Sub-phase 4: Persist + cache verified turns to `story_qa`
- **Files touched:** `agents/worker/main.py` (cache write inside the endpoint), `agents/qa/models.py` (cache model).
- **What ships:** Verified answers persisted to `story_qa` (service-role write — `story_qa` is a **content** table, no auth needed) with `qa_is_grounded` preserved, `qa_source_kind = 'rag_cached'` (legacy enum label retained to avoid a migration — now means "model-generated, verified, cached"), and citation outlet names into `qa_citation_outlet_names`; on a repeat of the same `(story_id, question)` the endpoint serves the cached verified row instead of re-running the LLM + verification. This answer cache **layers on top of** the SP2 per-story context cache. (`prototype-port-map.md` §7 mandates persisting turns; `supabase-schema.md` designed `story_qa` for exactly this parity/cache use.)
- **Definition of done:** Asking a question writes one `story_qa` row with the correct `qa_is_grounded` flag + citation outlet names; asking the identical question again returns the cached row **without** a second LLM/verification call (test asserts the cache-hit path is taken and the persisted grounded flag matches the live answer — Rule 9). `ruff` passes.
- **Dependencies:** Sub-phases 2 & 3.

## Phase-level definition of done
In the Detail view of a seeded story, a user types a question and receives a source-grounded answer with citation chips whose provenance traces to that story's grounding corpus; an off-source question yields the `⌀ CAN'T ANSWER FROM SOURCE` refusal — **never a fabricated fact**; verified turns persist to `story_qa` and repeat questions hit the cache. `/run-phase` validates: a grounded question and an off-source question against `s1` produce the two correct, visually-distinct states, and the off-source path never surfaces an answer bubble. **No vector store is provisioned or required.**

## Out of scope
- **Voice mode** (M3 mounts this same grounded brain over Gemini Live — `prototype-port-map.md` §7 "same brain… voice just streams"). The corpus loader + verification + endpoint contract are reused verbatim; voice only changes the transport.
- `player_signals` emission on `ask` — defers to **M3** (signals are `auth.uid()`-scoped; auth is M3; M2 stays auth-free). `story_qa` caching here is fine because `story_qa` is a service-role content table, not user-scoped.
- Multi-turn conversational memory beyond single-question grounding (the `conversation_id` field exists in `api-contracts.md`; M2's bar is single-turn grounded answers — multi-turn is an M3 stretch).
- **Vector retrieval / embeddings / a vector DB** — dropped (see re-scope above); reintroduce only if a story's corpus outgrows the context window (escape hatch).
- News ingestion / fetching full external article bodies (M1 ingestion).

## Open questions
1. **Grounding-corpus source (resolved).** The corpus is the story's `detail_chunks` (readable body — already seeded by M1, the single-source article per Decision #4) + `story_timeline` + the digest + any single-source body; `story_sources` provides the citation targets (outlet name + URL), not body text. This is small enough to load whole — confirmed sufficient for the M2 demo. Production breadth (full external article bodies) is M1 ingestion, out of scope here.
2. **Answer model + caching (`integrations.md`).** Confirm the answer LLM (Gemini, in-stack for TTS, vs Claude/OpenAI) and use its **prompt/context caching** for the per-story corpus block. No embedder/vector-dimension decision is needed anymore (retrieval dropped).
3. **Endpoint hosting (M1 infra).** The grounded endpoint runs server-side (Python worker FastAPI → Vercel/Railway/Fly, reached from the static SPA over HTTPS) to hold the LLM key + run verification. Confirm the worker is deployed by M1, or stand up a minimal endpoint as part of SP2. (Still required — this is the one remaining infra prereq after dropping Pinecone.)

## Self-critique

**Product lens:** PASS. Completes M2's "True when": *ask a question answered from the source*. This is the brief's **moat** (interrogate-in-place) and its zero-tolerance accuracy guardrail (Open Q5 / Decision #5), realized in SP2's verification + refusal contract. The re-scope changes nothing user-facing — same citation chips, same refusal card, same contract. No scope creep — voice, `player_signals`, and multi-turn memory stay deferred to M3.

**Engineering lens:** PASS — simpler than before. Dropping the vector layer matches the data shape (a 3-paragraph corpus needs no retrieval) and the "limit new code / don't blend where TLDW conflicts" directive (Rule 7): we keep TLDW's `chat/*` + `verification.py` (the parts that earn their keep) and skip `rag/*` (the part that doesn't, for a tiny per-story corpus). Every DoD is fresh-context-checkable (scope-only-this-story test, grounded-vs-refusal + provenance test, refusal-renders-no-answer test, cache-hit test). SP4 (cache) consumes the SP2 `QuestionAnswer` shape — it does not cement an API shape early. SP1 (corpus load) and SP2 (answer/verify) are genuinely distinct.

**Risk lens:** PASS (mitigated). **File boundary:** no two sub-phases edit the same file; SP3 edits Phase 2's `StoryDetail.tsx` but 2b runs sequentially after Phase 2's commit. **Reversibility:** there is **no shared external index to keep clean** anymore (Pinecone dropped) — the only writes are `story_qa` cache rows, which are deletable content rows. **Hallucination risk** (the trust-critical surface) is *reduced*: the model sees the whole source, so there is no retrieval-miss failure mode, and verification still gates the claim. **Cost/latency:** bounded by the small corpus + prompt caching; assert the corpus char/token budget in SP1 so an unexpectedly large story can't blow the window silently (fail loud, Rule 12).

**Irreversible sub-phases:** none destructive. SP4 writes `story_qa` content rows (deletable). No vector writes, no migrations (those are M1), no public-API-locking changes.
