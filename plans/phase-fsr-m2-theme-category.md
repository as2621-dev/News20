## Goal
A story's news category is derived from its GDELT `V2Themes` via a pure, deterministic, fail-loud `theme → FeedCategory` whitelist over all 8 topic categories (`ai, geopolitics, business, environment, politics, tech, sport, arts`), and that theme is carried from GKG ingestion through to the ingestion-time `story_interests` tag that `ranking.py::assign_category` already reads — replacing keyword-inherited category. `assign_category` itself is unchanged (it already picks the lowest-`match_depth` tag correctly); the fix is to the *tag a story gets*.

## Context the sub-agents need (verified facts — do not re-derive)
- **Category source of truth:** `agents/pipeline/categories.py` — the 8 topic roots are `FeedCategory` minus the two source axes (`youtube`/`x`). `TOPIC_CATEGORIES` lists exactly the 8. `DEFAULT_CATEGORY = "arts"` (the long-tail fallback). The PRD/ranking-spec say "5 folded categories" but the **live code is the 8-root model** (Rule 7 — pick the more recent/tested: use the 8 roots in `categories.py`). Flag this divergence; do not implement the stale 5-fold.
- **GKG SELECT today** (`agents/ingestion/adapters/gdelt_bigquery.py`, `_BATCH_SQL`) selects `V2Persons, V2Organizations, V2Locations` only — **`V2Themes` is NOT selected** (net-new). The `raw`→`base`→`matched`→`ranked` CTE chain and the `@interest_terms` STRUCT-array parameter binding are the only query surfaces; adding `V2Themes` means a new SELECT column in `raw`, carried through `base`/`matched`/`ranked` to the final projection, plus a new field on the row dict in `_rows_to_candidates`.
- **How category actually lands:** `assign_category(story_id, tags_by_story, interest_nodes)` (`agents/pipeline/stages/ranking.py`) takes the story's `story_interests` tags, picks the **lowest `match_depth`** tag, maps its interest **slug** → category via `category_for_slug`. So a story's category == the category of its most-specific `story_interests` tag. To make category theme-derived, the **theme must produce/override the tag the story carries** at ingest, NOT call into `assign_category`.
- **Tagging path:** `interest_keyed_pipeline.ingest_active_interests` → `merge_story_tags(story_id, canonical_matched_interest_ids, interest_nodes)` (`agents/ingestion/ancestor_tagging.py`) builds the `story_interests` payloads from the *matched interest ids* (keyword coincidence — the bug). `CandidateStory` (`agents/ingestion/models.py`) is the carrier from adapter → pipeline; it has no theme field today (net-new).
- **Sandbox is offline:** GDELT egress is 403, no BigQuery/DB creds. Every DoD here is a unit test over fixtures; the live GKG pull is `LIVE-E2E (deferred)`.
- **Test conventions:** mirror `tests/agents/pipeline/test_categories.py` and `tests/agents/ingestion/test_gdelt_bigquery_adapter.py`; mock at the boundary (no live BQ). The `_BATCH_SQL` string is a static artifact — assert substrings against it, do not execute it.

## Open questions
1. **Tag-wiring mechanism (the one real design decision):** `assign_category` resolves a *slug → category* via an `interest_slug`, but the theme whitelist yields a *category* directly. To make the category-determining (lowest-depth) tag theme-derived without touching `assign_category` or the schema, SP3 must either (a) carry the theme-derived `FeedCategory` on the canonical story / tag payload and have the categorization read it (cleanest, but `assign_category` reads slugs — may need a representative interest-slug per category, e.g. the root slug `ai`/`sport`/… which `category_for_slug` already maps identity), or (b) inject a synthetic depth-0 `story_interests` tag pointing at the category-root interest. **Recommendation:** (a)-via-root-slug — every category root slug already round-trips through `category_for_slug` identity, so a depth-0 tag on the category-root interest yields the theme category with zero schema/`assign_category` change. SP3 must confirm a category-root interest id exists in `interest_nodes` for each of the 8 roots; if not, that's the escalation. **`/run-phase` should resolve this in SP3's plan step before coding.**
2. **Whitelist coverage breadth:** the seed whitelist need only cover *representative* theme codes per category for the offline DoD (the PRD flags exhaustive theme curation as a tuning task, not a planning gate). Full GDELT theme-taxonomy curation is a LIVE-E2E follow-up, not an M2 blocker.
3. **Cross-milestone dependency:** none blocking. M2 is independent of M1 (catalog) and M3 (importance). It shares the `categories.py` 8-root taxonomy (read-only) and the `CandidateStory`/`CanonicalStory` models with M3/M4 — SP2/SP3 add fields (`candidate_themes`, `canonical_themes`) additively, so no conflict with concurrent milestones that only read those models.
