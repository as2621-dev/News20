This is a **PROMPT-ONLY** milestone — refine existing prompts; **no new ingestion, no schema, no new pipeline stage** (PRD Decision #10, Implementation Decision "Summaries").

## Goal
When this phase is done, a **long-form** followed-source item (a `youtube.com` video/podcast) is summarized as a **key-points** summary and a **short-form** followed-source item (an `x.com` tweet/short clip) is summarized **tightly** — the right prompt template is selected **in code** from the story (Rule 5), verified by offline unit/snapshot tests on fixtures (no live LLM call); and the revamp-affected reference docs + specs describe the new long-vs-short summary model.

## Key grounding facts (read before starting — verified in code, do not re-derive)
- **The long/short signal is already in the data and is deterministic** (Rule 5 — code selects the template, the model does not classify). A followed-source story's `canonical_primary_outlet_domain` is `youtube.com` (long-form video/podcast) or `x.com` (short-form/tweet); `agents/ingestion/dedup.py::is_source_origin_domain` + `SOURCE_ORIGIN_DOMAINS = {"youtube.com", "x.com"}` already encode this. **News (non-source) stories are neither** and must keep today's exact prompt behavior (regression guard).
- The two prompts that shape a source item's summary are: `DIGEST_SCRIPTING_PROMPT` (audio digest, in `agents/pipeline/prompts.py`, used by `agents/pipeline/stages/scripting.py`) and the `key_points` instruction inside `DETAIL_ENRICHMENT_PROMPT` (in `agents/pipeline/prompts.py`, used by `agents/pipeline/stages/detail_enrichment.py::_build_key_points` / `run_detail_enrichment`).
- Both stages build their system prompt via a local `_build_system_prompt(...)` that `.replace()`s `{PLACEHOLDER}` slots — the established templating convention (Rule 11). The long/short variant text must be injected through that same mechanism, NOT by branching into a second prompt module.
- Stages are invoked from `agents/pipeline/orchestrator.py` with a `CanonicalStory` already in hand (`run_single_source_scripting`, `run_detail_enrichment`), so the `story` needed to derive the mode is already available at every call site — no signature ripple beyond an optional, defaulted parameter.
- LLM is **mocked at the `LLMClient` boundary** in every test (`make_llm_client` fixture, `tests/agents/pipeline/test_scripting.py` / `test_detail_enrichment.py`). DoD assertions are on **the system prompt string actually sent** and on **which template was selected/interpolated** — no live call, no cost.

## Sub-phases

### Sub-phase 1: Add a code-side summary-mode selector + long/short prompt variant constants
- **Files touched:** `agents/pipeline/prompts.py` (add variant constants/blocks), **NEW** `agents/pipeline/summary_mode.py` (tiny pure selector), `tests/agents/pipeline/test_summary_mode.py` (NEW).
- **What ships:** A pure function `summary_mode_for(story) -> "long" | "short" | "news"` that maps `youtube.com → "long"`, `x.com → "short"`, everything else → `"news"` (delegating to `is_source_origin_domain` so the domain set stays single-sourced, Rule 7); plus the **variant instruction strings** in `prompts.py` that the two stages will interpolate — a `KEY_POINTS_LONG` block ("draw out the key points / substance of a 90-minute piece") and a `KEY_POINTS_SHORT` block ("tight summary; do not pad a tweet into filler"), each a distinct, asserts-different-text constant. No prompt is wired into a stage yet — this sub-phase only ships the selector + the building blocks.
- **Definition of done:** `test_summary_mode.py` asserts `summary_mode_for` returns `"long"` for a `youtube.com` story, `"short"` for an `x.com` story, `"news"` for a generic outlet domain (fixture `CanonicalStory`s only); a structural test asserts the long and short variant constants are **non-empty and differ** (long mentions key-points/bulleted-substance, short mentions tight/no-padding) — encodes WHY (Rule 9): a regression that collapsed the two shapes to one string fails here.
- **Dependencies:** none.

### Sub-phase 2: Select the long/short summary template in scripting
- **Files touched:** `agents/pipeline/stages/scripting.py` (`_build_system_prompt` + a defaulted `summary_mode`/derive-from-`story` path), `tests/agents/pipeline/test_scripting.py` (add cases).
- **What ships:** `run_single_source_scripting` / `_build_system_prompt` derive the mode via `summary_mode_for(story)` and interpolate the long vs short summary-shape instruction (from SP1's constants) into `DIGEST_SCRIPTING_PROMPT`; a **news** story interpolates the existing/neutral guidance so its prompt is **byte-for-byte unchanged** (regression guard). Single-source rule, length budget, persona contract, and JSON output contract are untouched.
- **Definition of done:** new `test_scripting.py` cases assert the **system prompt actually sent to the mocked client** (a) contains the long-form key-points shaping when `story.canonical_primary_outlet_domain == "youtube.com"`, (b) contains the tight-summary shaping (and NOT the long-form block) when `== "x.com"`, and (c) is **identical to today's prompt** for a news-domain story — asserting template selection + interpolation, no live call. Encodes WHY (Rule 9): selecting the wrong shape, or regressing the news path, fails.
- **Dependencies:** Sub-phase 1 (uses its selector + variant constants).

### Sub-phase 3: Select the long/short key-points template in detail enrichment
- **Files touched:** `agents/pipeline/stages/detail_enrichment.py` (`_build_system_prompt` / `run_detail_enrichment` thread the mode into the `key_points` instruction), `agents/pipeline/prompts.py` (make the `key_points` line of `DETAIL_ENRICHMENT_PROMPT` accept the SP1 long/short variant slot), `tests/agents/pipeline/test_detail_enrichment.py` (add cases).
- **What ships:** The detail enrichment's `key_points` instruction is interpolated long-vs-short by `summary_mode_for(story)` — a long-form source item asks for fuller key-points coverage, a short-form item asks for a tight set; **news stories keep today's exact 5-key-point instruction** (regression guard, and the "exactly 5" `_build_key_points` invariant + the numeric-grounding gate are untouched).
- **Definition of done:** new `test_detail_enrichment.py` cases assert the **system prompt sent to the mocked client** carries the long-form key-points shaping for a `youtube.com` story and the tight shaping (not the long block) for an `x.com` story, and is **unchanged for a news story**; the existing "exactly 5 key points" + ungrounded-number-dropped tests still pass (no regression to the trust gate). No live call. Encodes WHY (Rule 9): wrong key-points shape OR a regressed grounding/5-point invariant fails.
- **Dependencies:** Sub-phase 1 (selector + constants).
