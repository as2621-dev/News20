# Phase FSR-M1: Catalog clusters + no-dup resolver

**Milestone:** M1 — Catalog clusters + no-dup (data + content-ops) — `plans/prd.md` §Milestones
**Status:** Not started
**Estimated effort:** M

## Goal
A `source_clusters` + `source_cluster_members` schema exists over the existing `content_sources`/`personalities` catalog, an editorial seed of named clusters per category is authored, and a pure resolver returns — for one of the 8 categories — its clusters with ordered members **honoring the no-dup rule** (a followable personality's own YouTube/X handles are excluded from the rendered individual rows). Everything here is **offline-verifiable** (migration is a static `.sql` artifact + an apply-time assertion script; the resolver + no-dup logic are pure functions unit-tested against fixture catalog rows). This phase holds the revamp's **riskiest assumption — catalog quality** — so the de-risking check is fixture-based and runnable in this credential-less sandbox.

## Why this phase exists
M1 is the foundation the whole source-first thesis stands on: onboarding bulk-select (M6) cannot exist without a cluster grouping, and "no randoms / no duplicates" is the trust contract. Per PRD Decision #6/#7 a cluster is **net-new** (NOT an `archetype`, NOT a `persona`) and the no-dup rule is load-bearing. Schema + seed + resolver are the lowest-risk slice: data and a pure function first, live coverage later.

## Context the sub-agents need (verified — do NOT re-derive)
- **Reuse, do not re-create.** `content_sources` (cols: `source_id`, `content_source_type` enum `youtube_channel|podcast|x_account|personality`, `external_id`, `source_name`, `topic_tags text[]`, `popularity_score numeric`, `is_curated`) and `personalities` (cols: `personality_id`, `display_name`, `aliases text[]`, `youtube_channel_ids text[]`, `topic_tags text[]`, `popularity_score`) already exist from **`supabase/migrations/0009_content_sources.sql`** (skim it). M1 adds ONLY the two cluster tables — it does not touch 0009 objects.
- **Migration numbering.** Latest on disk is **`0021_taxonomy_8_roots_backfill.sql`** → this phase's migration is **`0022`**. Additive / forward-only (no down-migration), mirroring 0009's house style.
- **The 8 cluster categories** are the topic roots in `agents/pipeline/categories.py` `TOPIC_CATEGORIES`: `ai · geopolitics · business · environment · politics · tech · sport · arts`. `youtube`/`x` are source *axes*, NOT cluster categories — `cluster_category` ∈ the 8 topic roots only. This is the same vocabulary as `content_sources.topic_tags[0]`.
- **No-dup mechanism (verified).** A `personality` bundles its handles via `personalities.youtube_channel_ids text[]` (YouTube channel ids → matched against `content_sources.external_id` WHERE `content_source_type='youtube_channel'`) and `personalities.aliases text[]` (X handles → matched against `content_sources.external_id` WHERE `content_source_type='x_account'`). When a personality card is shown, those individual `content_sources` rows are excluded from the grid AND from any cluster's rendered members.
- **RLS tier (mirror 0009).** Cluster tables are **public-read** (anon onboarding reads them), service-role writes only — identical to `content_sources` (`for select using (true)`; no write policy).
- **Offline-test prior art to mirror.** Migration ships with an apply-time assertion script under `supabase/tests/` exactly like **`supabase/tests/0007_entity_registry_assertions.sql`** (the live apply is the LIVE-E2E residual; the offline gate is the migration parsing + structural assertions in a pure test). The resolver is a pure Python module + pytest, mirroring `agents/pipeline/categories.py` (pure, no DB/clock/network) and its test `tests/agents/pipeline/test_categories.py`; fixture/mocking conventions per `tests/scripts/seed_catalog/test_seed_catalog.py`.
- **Category resolution for dedup/tagging** reuses `agents/pipeline/categories.py::category_for_slug` / `TOPIC_CATEGORIES` — do not author a second category map.

## Sub-phases

### Sub-phase 1: Migration 0022 — `source_clusters` + `source_cluster_members` schema
- **Files touched:** `supabase/migrations/0022_source_clusters.sql`, `supabase/tests/0022_source_clusters_assertions.sql`.
- **What ships:** an additive, forward-only migration adding:
  - `source_clusters` — `cluster_id uuid pk`, `cluster_slug text not null unique`, `cluster_label text not null`, `cluster_category text not null` (constrained to the 8 topic roots via a `check (cluster_category in ('ai','geopolitics','business','environment','politics','tech','sport','arts'))`), `cluster_sort_order smallint not null default 0`, `is_curated boolean not null default true`, `cluster_created_at timestamptz not null default now()`.
  - `source_cluster_members` — `cluster_member_id uuid pk`, `cluster_id uuid not null references source_clusters(cluster_id) on delete cascade`, an **exactly-one-of** member ref: `source_id uuid references content_sources(source_id) on delete cascade` XOR `personality_id uuid references personalities(personality_id) on delete cascade` (a `check` that exactly one is non-null), `member_sort_order smallint not null` (the ordered position), `member_created_at timestamptz`, plus a partial-unique guard so the same `(cluster_id, source_id)` / `(cluster_id, personality_id)` pair can't be inserted twice.
  - Indexes: `source_clusters(cluster_category)`, `source_cluster_members(cluster_id, member_sort_order)`.
  - RLS: public-read on both tables (`for select using (true)`), no write policy (service-role only) — mirror 0009.
  - The companion `supabase/tests/0022_source_clusters_assertions.sql` (apply-time, `\set ON_ERROR_STOP on`, every check `raise exception` on violation — mirror 0007) asserting: both tables + the category check exist; a trial insert of a cluster + one `source_id` member and one `personality_id` member resolves all FKs (then ROLLBACK); the exactly-one-of check rejects a member with both refs null and a member with both set; anon `SELECT` on `source_clusters` succeeds.
- **Definition of done (OFFLINE):** a pure parse/structural test (`tests/supabase/test_migration_0022_source_clusters.py` or extend the seed test dir) reads `0022_source_clusters.sql` as text and asserts, **without a DB**: both `create table` statements present with the named columns; `cluster_category`'s 8-value `check` constraint lists exactly the 8 topic roots from `categories.TOPIC_CATEGORIES` (assert set-equality against the imported tuple, so the test fails if a root is dropped/added); the exactly-one-of member `check` is present; both `enable row level security` + a public-read `for select using (true)` policy and NO `for insert/update/delete` policy on each table; the migration number 0022 is unused on disk before this file. ⚠ irreversible (forward-only, additive). **LIVE-E2E residual (deferred):** `supabase db push` of 0022 + running `0022_source_clusters_assertions.sql` against the live DB.
- **Dependencies:** none.

### Sub-phase 2: Pure cluster + no-dup resolver
- **Files touched:** `agents/catalog/cluster_resolver.py`, `agents/catalog/__init__.py` (create if absent), `agents/catalog/models.py` (Pydantic `CatalogSourceRow`, `PersonalityRow`, `ClusterRow`, `ClusterMemberRef`, `ResolvedCluster`).
- **What ships:** a **pure** function `resolve_category_clusters(category, clusters, members, sources, personalities) -> list[ResolvedCluster]` (no DB/network/clock) that, given fixture rows: (a) filters clusters to `cluster_category == category`, ordered by `cluster_sort_order`; (b) for each cluster, resolves members in `member_sort_order`, expanding `source_id`/`personality_id` refs to display rows; (c) **applies the no-dup rule** — when a personality is present (as a personality member anywhere in the category, OR as a `content_sources` row that a present personality bundles), exclude the `content_sources` YouTube rows whose `external_id ∈ that personality.youtube_channel_ids` and the `x_account` rows whose `external_id ∈ that personality.aliases`, from both the grid and every cluster's rendered members; (d) **skips** members whose referenced row is missing/un-curated (don't error); (e) **drops empty clusters** (don't surface) after dedup; (f) dedups a row that appears in two clusters of the same category (allowed in data; deduped at selection — return each underlying followable once with a stable "first cluster wins" or documented tiebreak). Category typing reuses `categories.TOPIC_CATEGORIES`.
- **Definition of done (OFFLINE):** `tests/agents/catalog/test_cluster_resolver.py` (pytest, fixture rows, no DB) asserts the **business rules, not shape** (Rule 9): (1) **cluster membership + order** — a 3-member cluster returns its members in `member_sort_order`; (2) **empty cluster** — a cluster whose only member was un-curated/deleted is NOT in the output; (3) **personality dedup** — a personality bundling a YouTube channel id and an X handle that ALSO exist as `content_sources` rows causes those two rows to be absent from the grid and from any cluster's members, while the personality card appears once (WHY: PRD Decision #7 — a person shown twice breaks the trust contract); (4) **multi-category row** — a source tagged to two categories appears under each category call, deduped within a category; (5) **member in two clusters of one category** — returned once at selection, no crash. Each test name/docstring encodes the WHY. `ruff` clean.
- **Dependencies:** Sub-phase 1 (for the table/column contract — but the resolver consumes Pydantic fixture rows, not a live DB, so it can be authored from THIS file's column spec in parallel; marked sequential only for the contract-of-record).

### Sub-phase 3: Editorial cluster seed (named clusters per category)
- **Files touched:** `supabase/seed/source_clusters.sql`.
- **What ships:** a hand-authored, **idempotent** (`on conflict (cluster_slug) do update` / `on conflict do nothing` for members, mirroring `supabase/seed/archetypes.sql` style) seed of named clusters across the 8 categories over EXISTING `content_sources`/`personalities` rows — e.g. AI: "Leading AI-lab researchers", "AI founders", "AI journalists"; with ordered `source_cluster_members` referencing real catalog rows by their `(content_source_type, external_id)` / personality `display_name` resolved to ids via sub-selects (so the seed does not hardcode uuids). Each cluster carries `cluster_category` ∈ the 8 roots and `cluster_sort_order`. At least 2 clusters per category where the existing catalog supports it; categories with a thin catalog get ≥1 and are listed (fail-loud, not silently empty).
- **Definition of done (OFFLINE):** a pure parse/structural test (`tests/seed/test_source_clusters_seed.py`) reads `source_clusters.sql` as text and asserts, **without a DB**: every `insert` into `source_clusters` carries a `cluster_category` ∈ the 8 `TOPIC_CATEGORIES` (set-membership assertion, imported from `categories.py`); the seed is idempotent (`on conflict` present on every insert); every category root appears at least once OR is explicitly listed in a `-- THIN CATALOG:` comment line (so a gap is loud); member inserts reference rows via sub-select on `(content_source_type, external_id)` / `display_name`, never raw uuids. **LIVE-E2E residual (deferred):** applying the seed against the live DB and confirming real per-category coverage (the #1 risk's live check — real catalog rows actually resolve).
- **Dependencies:** Sub-phase 1 (FKs to the new tables). Disjoint files from SP2 → schedulable in parallel with SP2.

### Sub-phase 4: Category → ordered-clusters query + fixture integration test
- **Files touched:** `agents/catalog/cluster_query.py`, `tests/agents/catalog/test_cluster_query_fixture.py`, `tests/agents/catalog/fixtures/catalog_fixture.json`.
- **What ships:** the thin query/assembly layer `clusters_for_category(category, *, repo) -> list[ResolvedCluster]` that loads catalog + cluster rows from a **repository interface** (a `CatalogRepo` Protocol with one in-memory fixture implementation here; the live Supabase implementation is the LIVE-E2E residual) and pipes them through SP2's `resolve_category_clusters`. Ships a committed `catalog_fixture.json` (a small but realistic multi-category catalog: a personality bundling handles that also exist as source rows, an empty cluster, a multi-category source, a member shared across two clusters) so the end-to-end M1 behavior is provable offline.
- **Definition of done (OFFLINE):** `test_cluster_query_fixture.py` loads `catalog_fixture.json` into the in-memory repo and asserts the **milestone-level** contract end-to-end: calling `clusters_for_category('ai')` returns AI's clusters with ordered members AND the personality-bundled source rows excluded (the no-dup rule holds through the full path, not just the resolver unit) — this is M1's offline de-risking check. Plus: a category with no clusters returns `[]` (not a crash); the fixture's empty cluster is absent. WHY encoded per test. `ruff` clean.
- **Dependencies:** Sub-phases 1, 2, 3 (consumes the resolver, the schema contract, and mirrors the seed's shape in the fixture).

## Phase-level definition of done
Migration `0022_source_clusters.sql` exists as a valid, structurally-asserted additive artifact (+ its apply-time assertion script); a pure `resolve_category_clusters` resolver passes unit tests for membership/order, empty-cluster, personality-dedup, multi-category, and shared-member cases; an idempotent editorial `source_clusters.sql` seed is authored and structurally validated; and `clusters_for_category(category)` returns ordered clusters with no-dup-honored members against a committed catalog fixture. **Validated OFFLINE by:** the 0022 parse/structural test (8-root check ↔ `categories.TOPIC_CATEGORIES`), the resolver unit suite, the seed structural test, and the fixture integration test — all runnable with no DB and no credentials. **LIVE-E2E residual (deferred):** `db push` of 0022 + assertion script, applying the seed, and confirming real per-category catalog coverage against the live Supabase DB.
